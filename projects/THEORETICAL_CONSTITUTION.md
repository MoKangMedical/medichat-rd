## 序言：为什么需要理论宪法？

大多数创业者从"做什么"开始。我们从"为什么这么做"开始。

理论宪法回答三个根本问题：
1. **认知论**：我们如何知道什么值得做？（鉴别诊断式调研）
2. **方法论**：我们如何把事情做成？（Harness + 知识驱动 + 达尔文棘轮）
3. **价值论**：我们为什么存在？（从治愈身体到构建思想基础设施）

没有理论的项目是碰运气。有理论的项目是做实验。

## 第一卷：认知论 — 我们如何思考

### 1.1 鉴别诊断式决策框架

**来源**：临床医学的鉴别诊断方法论
**核心隐喻**：面对一个症状（市场机会），不急于确诊（立项），而是系统性排除不可能的选项。

```
症状识别 → 候选假设列表 → 系统性排除 → 确诊 → 治疗方案
    ↓              ↓              ↓          ↓         ↓
市场信号 → 可能的方向 → 调研排除 → 选定赛道 → 技术方案
```

#### 五步流程

1. **症状收集**：市场出现了什么信号？（用户痛点/技术突破/政策变化）
2. **候选假设**：列出所有可能的应对方向（≥3个）
3. **鉴别排除**：对每个方向做系统性调研
   - 海外有没有标杆？（有→研究架构精华）
   - 国内有没有人在做？（有→差异化在哪）
   - 技术可行性？（能不能做出来）
   - 商业可行性？（能不能卖出去）
4. **确诊**：选定方向，排除其他
5. **治疗方案**：技术架构 + 商业模式 + 迭代计划

#### 与直觉决策的区别

| 维度 | 直觉决策 | 鉴别诊断式决策 |
|------|---------|--------------|
| 速度 | 快 | 慢（但更准） |
| 依据 | 个人经验 | 系统调研 |
| 风险 | 高（认知偏差） | 低（结构化排除） |
| 可复现 | 否 | 是（过程可追溯） |

#### 应用案例

- **MediChat-RD**：症状（罕见病诊断难）→ 假设（AI辅助/社区/工具平台）→ 排除（社区已有，工具太窄）→ 确诊（AI诊断平台）→ 方案（Harness + 多Agent）
- **窄门**：症状（AI产品同质化）→ 假设（垂直优化/新交互/思想层）→ 排除（垂直已是红海）→ 确诊（思想基础设施）→ 方案（知识驱动 + 大师引导）

### 1.2 知识驱动开发方法论

**来源**：窄门项目实践提炼
**核心命题**：先构建理论知识库，再基于知识基座进行产品开发。

#### 传统开发 vs 知识驱动开发

```
传统：需求 → 设计 → 编码 → 测试 → 上线
         ↓
      用户说什么做什么

知识驱动：理论体系 → 知识库 → 架构设计 → 编码 → 测试 → 上线
              ↓
          先想清楚"为什么"，再做"怎么做"
```

#### 三层结构

```
┌──────────────────────────────┐
│  Layer 3: 产品层              │  ← 用户看到的
│  （前端/交互/API）            │
├──────────────────────────────┤
│  Layer 2: 工程层              │  ← 开发者看到的
│  （代码/架构/部署）           │
├──────────────────────────────┤
│  Layer 1: 知识层 ⭐ 根基      │  ← 最先构建，最后被看到
│  （理论/哲学/方法论）         │
└──────────────────────────────┘
```

#### 为什么知识层必须先建？

1. **减少返工**：理论不清晰，代码写了也得推翻
2. **差异化来源**：技术可以抄，知识体系抄不了
3. **可扩展性**：知识层决定产品能走多远
4. **护城河深度**：代码壁垒 < 架构壁垒 < 知识壁垒

#### 适用范围

不是所有项目都需要知识驱动。判断标准：
- ✅ 适合：思想型产品、平台型产品、需要长期积累的领域
- ❌ 不适合：快速验证的MVP、纯工具型产品、短期项目

## 第二卷：方法论 — 我们如何做事

### 2.1 Harness理论 ⭐

> **详见**：`HARNESS_THEORY.md`
> **核心命题**：在AI领域，Harness（环境设计）比模型本身更重要。
> **一句话**：模型是通用的，Harness是私有的。

#### 速览

```
七层生态：
L7 业务价值层    ← 用户付费的
L6 Harness设计层 ← 我们的护城河 ⭐
L5 Agent框架层
L4 工具集成层
L3 上下文管理层
L2 推理优化层
L1 模型层        ← 大家都有的
```

#### 四项目应用

| 项目 | Harness类型 | 护城河来源 |
|------|------------|-----------|
| MediChat-RD | 诊断Harness | 鉴别诊断决策树 + 多学科会诊编排 |
| MediSlim | 健康管理Harness | 全流程服务编排 + 数据闭环 |
| MediPharma | 药物发现Harness | Pipeline架构 + 多维度评分 |
| 窄门 | 思想Harness | 大师引导者 + 进化金字塔 |

### 2.2 达尔文棘轮协议 ⭐

**来源**：Karpathy autoresearch → 达尔文棘轮机制
**核心命题**：用进化论的"自然选择"机制驱动项目质量持续提升，而非一次性评审。

#### 五铁律

1. **单一资产铁律**：每轮只评审一个核心资产（不是整个项目）
2. **双重评估铁律**：结构完整性（60分）+ 效果质量（40分）= 100分制
3. **棘轮铁律**：改进后得分 ≥ 改进前得分（只进不退）
4. **独立评分铁律**：每轮由独立评审打分（不自评）
5. **人在回路铁律**：关键决策需要人类确认

#### 运行机制

```
R1: 基线评估 → 发现问题 → 改进 → 评分
R2: 独立评审 → 发现新问题 → 改进 → 评分（≥ R1）
R3: 深度审查 → 战略级改进 → 评分（≥ R2）
R4: 最终验证 → 产出稳定版
R5: 闭环确认 → Demo + 视频 + 推送
```

#### 实战数据

| 项目 | 起始分 | R5终分 | 提升 | 改进次数 |
|------|--------|--------|------|---------|
| MediSlim | 55 | 98 | +43 | 6 |
| 天眼 | 79 | 91 | +12 | 5 |
| Harness框架 | 75 | 100 | +25 | 5 |
| MediChat-RD | 62 | 88 | +26 | 5 |
| MediPharma | 72 | 85 | +13 | 5 |
| DrugMind | 70 | 82 | +12 | 5 |
| **六项目均值** | **70.8** | **88.7** | **+17.9** | **27次改进，0回滚** |

#### 达尔文棘轮 vs 传统Code Review

| 维度 | Code Review | 达尔文棘轮 |
|------|------------|-----------|
| 频率 | 每次PR | 多轮迭代 |
| 范围 | 单文件/单功能 | 整体资产 |
| 标准 | 功能正确 | 结构+效果双维 |
| 进化 | 无累积机制 | 棘轮（只进不退） |
| 产出 | 通过/不通过 | 量化评分 + 改进轨迹 |

### 2.3 百川归海 — 项目生命周期理论

**来源**：MedRoundTable实践经验
**核心命题**：独立子项目成熟后，应归并入主产品线，如同百川归海。

#### 四阶段生命周期

```
探索期 → 爆发期 → 成熟期 → 归海期
  ↓         ↓         ↓         ↓
独立仓库  快速迭代  稳定运行  合并入主产品线
验证价值  建立壁垒  提炼精华  释放仓库资源
```

#### 归海标准

子项目可以归海当且仅当：
1. ✅ 核心功能已完成（非半成品）
2. ✅ 方法论已提炼（可复用的知识）
3. ✅ 主产品线有对应位置（有地方归）
4. ✅ 代码已打包/文档化（不丢失价值）

#### 归海 vs 淘汰

| 归海 | 淘汰 |
|------|------|
| 价值被继承 | 价值被丢弃 |
| 精华融入主产品 | 整体废弃 |
| 进化 | 死亡 |

**核心隐喻**：河流汇入大海后，河水不再以独立身份存在，但它构成了大海的一部分。

## 第三卷：价值论 — 我们为什么存在

### 3.1 红杉论点 — 从卖工具到卖结果

**来源**：Sequoia Capital 2024 论文
**核心命题**：下一代万亿美元公司是伪装成服务公司的软件公司。

#### 逻辑链条

```
大模型进步 → 服务成本下降 → 服务利润空间上升
    ↓
不再卖工具（SaaS），卖结果（Service）
    ↓
AI做90%运营，人类做10%决策
    ↓
边际成本趋零，规模效应无限
```

#### 我们的统一叙事

| 项目 | 卖什么（错误） | 卖什么（正确） |
|------|--------------|--------------|
| MediChat-RD | 诊断工具 | 诊断结果（症状→确诊报告） |
| MediSlim | 健康SaaS | 健康管理结果（评估→方案→效果） |
| MediPharma | AI平台 | 候选化合物（靶点→先导分子） |
| 窄门 | 哲学对话工具 | 思想成长结果（困惑→洞见→行动） |

⚠️ **铁律**：所有商业计划书、融资材料、对外叙事，必须包含此论点。

### 3.2 OPC一人公司理论

**来源**：小林医生的创业实践
**核心命题**：在AI时代，一个人可以运营一家完整公司。

#### OPC公式

```
OPC = 创始人（战略+决策） + AI Agent（执行+运营） + 外包团队（专业补充）
        10%                    80%                     10%
```

#### 为什么现在可能？

1. **AI执行能力**：代码/内容/分析/客服 → Agent可完成
2. **工具生态成熟**：GitHub/Vercel/Stripe → 基础设施即服务
3. **外包专业化**：设计/法务/财务 → 按需采购
4. **MIMO API无限额度**：零边际成本推理

#### OPC的边界

- ✅ 适合：技术产品、知识产品、数字服务
- ❌ 不适合：重运营、强监管、需要线下触达的行业
- ⚠️ 灰色区：医疗（需要人工兜底，但AI可做90%）

#### OPC vs 传统创业

| 维度 | 传统创业 | OPC |
|------|---------|-----|
| 团队 | 5-50人 | 1人 + N个Agent |
| 成本结构 | 固定成本高 | 边际成本趋零 |
| 决策速度 | 慢（需要协调） | 快（一个人决定） |
| 扩展方式 | 招人 | 加Agent |
| 极限规模 | 受人数限制 | 受Agent能力限制（持续提升） |

### 3.3 思想基础设施理论

**来源**：窄门项目
**核心命题**：AI产品最终竞争的不是功能，而是思想深度。

#### 产品矩阵的思想层级

```
Level 4: 思想基础设施（窄门）     ← 构建思维方式
Level 3: 效率基础设施（DrugMind） ← 构建工作方式
Level 2: 情感基础设施（念念）     ← 构建连接方式
Level 1: 身体基础设施（医疗AI）   ← 构建生存方式
```

#### 为什么思想层是终极护城河？

1. **功能可抄**：你做一个问诊机器人，别人一周能复刻
2. **架构可学**：你的Harness设计，别人研究后能模仿
3. **思想难抄**：你的哲学立场、引导方式、价值判断 → 这是灵魂

#### 窄门的设计哲学

好的AI不应有问必答，而应像苏格拉底一样追问引导。
- **不给答案，给问题**
- **不给安慰，给挑战**
- **不给依赖，给成长**

## 第四卷：元理论 — 理论之间的关系

### 4.1 统一架构图

```
                    ┌──────────────────────┐
                    │   价值论（Why）        │
                    │ 红杉论点 + OPC + 思想  │
                    └──────────┬───────────┘
                               │ 定义目标
                    ┌──────────▼───────────┐
                    │   认知论（What）       │
                    │ 鉴别诊断 + 知识驱动    │
                    └──────────┬───────────┘
                               │ 选择方向
                    ┌──────────▼───────────┐
                    │   方法论（How）        │
                    │ Harness + 棘轮 + 归海  │
                    └──────────┬───────────┘
                               │ 执行落地
                    ┌──────────▼───────────┐
                    │   项目矩阵（Do）       │
                    │ MediChat / Slim /      │
                    │ Pharma / 窄门 / ...    │
                    └──────────────────────┘
```

### 4.2 理论间的强化关系

```
Harness理论 ←──支撑──→ 红杉论点
    ↓                      ↓
（技术护城河）         （商业叙事）
    ↓                      ↓
达尔文棘轮 ←──验证──→ OPC理论
    ↓                      ↓
（质量保证）           （运营模式）
    ↓                      ↓
知识驱动 ←──根基──→ 思想基础设施
    ↓                      ↓
（开发方法论）         （终极护城河）
```

每个理论都不是孤立的。它们互相支撑，形成一个**自洽的理论网络**。

### 4.3 理论的进化机制

理论本身也需要达尔文棘轮：

```
实践反馈 → 理论修正 → 新实践验证 → 理论固化
    ↑                                  │
    └──────────────────────────────────┘
```

**每季度审查**：
1. 哪些理论假设被实践推翻了？
2. 哪些新现象无法被现有理论解释？
3. 哪些理论需要扩展/修正/淘汰？
